## 01. 配置文件用 YAML 不用 JSON(2026-04-24)

**状态**:采纳

**背景**:项目需要一种配置文件格式,用于任务的运行参数、字段声明和任务元信息。
修改者包括项目作者本人、未来接手者、与项目协作的 AI。配置文件需要被
启动器读写,也需要人类直接编辑。

**决策**:统一使用 YAML。

**考虑过的替代方案**:

- **JSON**:无注释支持,三个月后看不懂每个字段是干嘛的。排除
- **TOML**:Python 3.13 自带 `tomllib` 只能读不能写,启动器要可视化编辑配置
  需要写功能,排除
- **Python 字典文件(.py)**:能用但非标准,编辑器支持差,排除

**代价**:

- PyYAML 是外部依赖,需要搬运到目标环境一次
- YAML 的缩进敏感,对不熟悉的人容易出错(文档注释可缓解)

## 02. 启动器用 Tkinter 不用 PyQt/HTML(2026-04-24)

**状态**:采纳

**背景**:需要一个简单的 GUI 启动器,让用户可视化选择任务、编辑配置、看日志。
目标环境可能有依赖搬运困难。

**决策**:用 Tkinter。

**考虑过的替代方案**:

- **PyQt/PySide**:界面更漂亮,但依赖包大,搬运麻烦
- **HTML + 本地 http server**:灵活,但启动步骤多,违反最高原则(用户体验优先)
- **命令行菜单(bat 文件)**:更轻量,但无法满足"可视化编辑配置"需求

**代价**:

- Tkinter 界面偏丑
- 原生组件样式有限,但项目哲学接受"不调花哨样式",可接受

## 03. 启动器不 import 任务代码,只通过子进程+文件通信(2026-04-24)

**状态**:采纳

**背景**:启动器和任务之间需要通信——启动器要触发任务、读配置、捕获日志。
存在两种方式:import 任务代码作为模块调用,或起子进程调用独立脚本。

**决策**:只通过子进程和文件通信,启动器绝不 import 任务代码。

**考虑过的替代方案**:

- **启动器 import 任务的 runner 函数直接调用**:性能更好,但绑死了启动器和任务,
  一个任务挂了启动器也挂;任务无法独立命令行运行;调试困难

**理由对应哲学**:项目工程哲学 2.1 维度 B——可执行实体应与触发方式分离。

**代价**:

- 子进程启动慢,但任务执行时间远大于启动时间,可接受
- 通过文件和 stdout 通信比直接 import 多一层隔阂,但换来了可独立运行和调试方便

## 04. 每个任务三份 YAML 而不是一份(2026-04-24)

**状态**:采纳

**背景**:每个任务需要保存三类信息:元信息(给启动器看)、运行参数(给 runner 看)、
字段声明(给启动器生成编辑 UI)。三类信息读者不同、变化频率不同。

**决策**:拆成三份文件——task.yaml、config.yaml、config.schema.yaml。

**考虑过的替代方案**:

- **一份大 YAML 装所有**:改起来不方便,启动器显示任务列表要加载全部配置,
  违反"变化频率不同的东西应该分离"原则

**理由对应哲学**:项目工程哲学 2.1 维度 A——变化频率不同的东西应该分离。

**代价**:

- 新建任务要写三个文件,模板可缓解

## 05. 日志用 print 不用 logging(2026-04-24)

**状态**:采纳

**背景**:启动器要实时捕获任务的日志输出显示到 UI。

**决策**:任务用 `print` 输出,配合 `sys.stdout.reconfigure(line_buffering=True)`。

**考虑过的替代方案**:

- **logging 模块**:功能更强(handler/formatter/level),但默认带 buffer,
  启动器日志区会卡顿;配置复杂

**代价**:

- 日志没有内建的时间戳和级别
- 需要时间戳要自己 `print(f"[{datetime.now()}] ...")`
- 项目规模小,不需要 logging 的高级能力,可接受

## 本文档的维护规则

**追加新决策**:有新的非直觉设计决策时,按格式追加到本文档末尾。

**推翻旧决策**:被新决策替代时,修改旧决策的"状态"字段为"已推翻(被 NN 决策替代)"。
**不删除旧决策**——保留历史供未来回看。

**归档/删除旧决策**:如果某条决策已经成为"项目常识"(不再有人会问),
或者已经完全不适用于当前项目,可以移到本文档末尾的"归档区",或直接删除。
归档/删除前回看一次,确认该决策涉及的内容已经在其他文档里充分体现。

**编号**:追加时用下一个编号。推翻重写时保留原编号,不要重排。
