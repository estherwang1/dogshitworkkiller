# DogshitWorkKiller 设计哲学

本文档是整个项目的世界观和美学基准——回答"什么是好的",不回答"具体怎么做"。

**本文档适用范围**:整个 DogshitWorkKiller 项目(单个系统层面)。
跨系统集成、多项目协同等问题不在本文档范围内。

**文档的使用方式**:

- 遇到设计分歧,用哲学裁决方向
- 遇到规则冲突,按"最高原则 > 通用工程哲学 > 项目工程哲学 > 判断原则"裁决
- 具体怎么做,不在本文档——见编码规范和架构文档

**与其他文档的关系**:

- 哲学 = 世界观(本文档)
- 架构文档 = 图纸目录
- 编码规范 = 施工标准
- 决策日志 = 改造档案

哲学不引用任何其他文档。其他文档可以引用哲学。

---

## 第〇部分:最高原则

### 用户体验优先于代码优雅

当"更优雅的代码"和"更舒适的使用"冲突时,选后者。

代码是过程,体验是目的。用户使用时的一次不爽,比写代码时一百次不爽更重要。

这条原则**凌驾于本文档其他所有原则之上**。任何原则在特定场景下与它冲突,
以它为准。

**关于"用户"的适用范围**:此处的"用户"指所有与本项目交互的人,包括
项目作者自己、同事、未来接手者,也包括与项目协作的 AI。

## 第一部分:通用工程哲学

任何软件项目都适用的基础原则。本项目采纳 Python 和软件工程社区的共识。

### 1.1 DRY(Don't Repeat Yourself)

同一段逻辑出现两次视为巧合,三次则应抽象。

**关键区分**:是"逻辑重复"才抽象,不是"代码长得像"就抽象。
两段代码长得像但表达的是不同意图时,保持分开。

### 1.2 SRP(Single Responsibility Principle)

一个函数、一个模块只做一件事。

**判断方法**:能不能用一句简单的话描述它的作用。如果描述时出现
"并且"、"然后"、"同时",通常要拆。

### 1.3 YAGNI(You Aren't Gonna Need It)

不为"未来可能的需求"提前写代码。只实现当下真正用到的功能。

过度工程化的代价是维护成本,不是资产。

### 1.4 显式优于隐式

出自 Python 之禅(`import this`)。

- 函数参数写清楚,不依赖全局变量
- 配置字段写清楚,不靠约定俗成
- 异常类型写清楚,不用 `except Exception` 一把捞

## 第二部分:项目工程哲学

本项目特有的工程原则。

### 2.1 解耦优先,避免过度解耦【纲领】

项目长期可维护性的根基在于解耦——不同关注点互不干扰,改一处不影响其他处。

**解耦的五个维度**:

#### 维度 A:变化频率不同的东西应该分离

慢变的被快变的污染,快变的被慢变的束缚。

#### 维度 B:可执行实体应与触发方式分离

做事的东西和启动它的东西应该分开。实体本身能独立运行,
触发方式只是多一种调用途径。

#### 维度 C:单个失败应与整体流程分离

批量处理中,单次失败要被捕获并记录,不能让整个流程崩溃。
失败是常态,不是异常。

#### 维度 D:通用工具应与业务逻辑分离

通用能力(读文件、调 API)和业务逻辑(怎么评估规范、怎么分类邮件)
分开存储。

**判断方法**:这段代码搬到另一个完全不同的项目能不能直接用?
能→是通用工具;不能→是业务逻辑。

#### 维度 E:数据资产应与操作代码分离

prompt、schema、mapping 这类"数据"虽然以代码形式存在,但本质是资产。
它们应与操作它们的代码分开。

修改数据的人可能不是程序员,分开后改数据不改代码,降低门槛和风险。

---

**解耦的边界**:不追求形式干净

解耦的目的是**降低未来的维护成本**,不是追求形式上的"干净"。
当解耦带来的认知成本高于它减少的维护成本时,不要解耦。

**判断原则**:解耦应该让你**省心**,不是让你**省事**。
省心 = 未来维护容易;省事 = 当下少写代码。两者冲突时选省心。

### 2.2 文档服务于接手者,不是作者

写文档前问三个问题:

1. 三个月后的我打开,能不能快速回忆起项目状态?
2. 一个新人(或新对话的 AI)读完,能不能立刻上手?
3. 有没有具体步骤,还是只有空话?

**禁止写入文档的内容**:

- 空洞的口号或金句(最高原则是唯一例外,它是整个项目的纲领)
- 未来可能会有的东西
- 代码能直接看出的信息

**应该写入文档的内容**:

- 目录结构和每层职责
- 非直觉的设计决策及其原因
- 新增功能的标准步骤
- 已知的坑和解决方法

### 2.3 架构从实例中长出,不从想象中设计

此处"架构"指单个系统内部的架构(本项目内部的组织方式)。

不要为"未来可能的需求"提前设计架构。架构应该从**至少两个真实实例**
中提炼,不是凭空构造。

**过度设计的典型症状**:

- 设计了 N 层架构,实际只用到 2-3 层
- 定义了复杂的数据契约,却只有一方在用
- 预留了大量扩展点,每个都没被用过

**例外**:如果某个抽象**现在不加、后面加的成本会指数级上升**
(如数据库 schema、对外 API 格式),可以提前设计。
但要能明确说出"为什么不提前做成本会爆炸"。

## 第三部分:判断原则

文档不可能覆盖所有情况。遇到没明确规定的新问题时,用本节的思路决策。

### 3.1 何时抽到通用工具层?

**判断标准**:至少两个真实用例已经用到,且该能力本身不含业务。

- ✓ 两个场景都要读 YAML 配置 → 抽成通用工具
- ✓ 三个场景都要某种失败记录机制 → 抽成通用工具
- ✗ 只有一个场景用到 → 不抽(违反 YAGNI)
- ✗ 函数名含业务词汇 → 不抽(含业务就不是通用工具)

**反模式**:"我觉得以后可能会用到"是最常见的错误理由。
未来的事交给未来。

### 3.2 何时新增业务单元?

**判断标准**:业务本质不同,不是输入形态不同。

- ✓ 业务目标完全不同 → 不同业务单元
- ✗ 同一业务,只是输入形态不同 → 同一业务单元
- ✗ 同一业务,只是参数不同 → 同一业务单元(参数是配置)

**判断方法**:如果两件事的处理逻辑完全一样,就是同一业务单元;
如果逻辑要完全重写,是不同业务单元。

### 3.3 何时加新依赖?

**判断标准**:标准库和现有依赖确实无法满足,且新依赖能被部署到目标环境。

- ✗ 为写个 CLI 加依赖 → 标准库够用
- ✗ 为美化日志加依赖 → 基础输出够用,违反最高原则(不必要的复杂化)

新依赖进项目前必须更新依赖清单,并记录部署方式。

### 3.4 何时打破上面的原则?

**判断标准**:最高原则(用户体验优先)与其他原则冲突时。

**合理的打破**:

- 原则说"应该分离",但分离后用户使用反而更累 → 合回来,注释说明理由
- 原则说"不该提前抽象",但不抽象每次改要改五处 → 提前抽象,注释说明理由

**不合理的理由**:

- "觉得写规范麻烦"
- "当前不重要以后再说"

**执行要求**:每次打破原则都要**在代码注释或提交信息里简短说明理由**。
让未来的接手者知道这是有意识的决策,不是疏忽。

---

## 结尾

**原则的目的是让你省心,不是增加负担**。

如果某条原则每次都在拖慢你,说明它可能已经过时或本来就不对。
把它改掉,不要硬守。原则应该服务人,不是反过来。

文档也一样。如果文档变成了"写了但没人看"的负担,说明写得不对,该精简。
好文档是接手者主动愿意读的,不是被迫看的。
