# DogshitWorkKiller 设计哲学

本文档是整个项目的最高约束,优先级高于任何具体代码实现。
所有参与开发的人(包括未来的你、未来的 AI 协作者)都应先读此文档,再动代码。

文档的使用方式:
- 遇到设计决策,先翻这里找对应原则
- 遇到规范冲突,按"最高原则 > 通用规范 > 项目原则 > 项目契约"的顺序裁决
- 遇到文档没覆盖的情况,用"判断原则"一节的思路决策

**文档分层说明**:
- 第〇、一部分是任何 Python 项目都适用的通用内容
- 第二部分是本项目工程哲学,表达为抽象原则
- 第三部分是本项目具体契约,把抽象原则落地为可执行的边界约定
- 第四部分是判断原则,用于处理文档没覆盖的新情况

---

## 第〇部分:最高原则

### 用户体验优先于代码优雅

当"更优雅的代码"和"更舒适的使用"冲突时,选后者。

代码是过程,体验是目的。用户(自己或同事)使用时的一次不爽,
比写代码时一百次不爽更重要。

这条原则**凌驾于本文档其他所有规范之上**。任何规范在特定场景下
与它冲突,以它为准。举例:

- 规范说"配置与代码分离",但某个参数只在一处用、改起来要翻
  多层目录反而慢——这时硬编码一个常量更合适
- 规范说"函数要单一职责",但拆完用户要多点三次才能跑完任务
  ——这时合并回来更合适
- 规范说"不要提前抽象",但某个重复代码改一次要改五处、未来
  肯定还会改——这时提前抽象更合适

这不是为偷懒开后门,而是提醒:所有规范都是手段,用户体验才是目的。
每次打破规范时,应在代码注释或提交信息里简短说明理由。

## 第一部分:通用编码规范(L1 层)

任何 Python 项目都应遵守的基准线。本项目在此之上再加额外约束。

### 1.1 代码风格:遵循 PEP 8

- **缩进**:4 个空格,禁用 Tab
- **行宽**:不超过 120 字符(比 PEP 8 默认的 79 宽松,适应现代屏幕)
- **空行**:顶层函数和类之间空 2 行,类内方法之间空 1 行
- **运算符两侧空格**:`a = 1 + 2`,不是 `a=1+2`
- **逗号后空格**:`foo(a, b, c)`,不是 `foo(a,b,c)`

推荐使用 `ruff` / `black` 等格式化工具自动处理。

### 1.2 命名规范

- **变量、函数、模块名**:`snake_case`
- **类名**:`PascalCase`
- **常量**:`UPPER_CASE_WITH_UNDERSCORES`
- **私有成员**:前缀单下划线 `_private_helper`(不用双下划线,除非真的需要名称改写)
- **中文**:注释、字符串、文档可以用中文;**变量名、函数名、文件名必须用英文**

命名要描述"是什么"或"做什么",不要用 `data1`、`tmp`、`process()` 这类无信息量的名字。

### 1.3 注释与文档

**Docstring 风格:Google 风格**

所有对外接口必须有 docstring。内部辅助函数可省,但若逻辑复杂仍建议写。

格式示例:

```python
def parse_document(path: str) -> list[dict]:
    """解析文档,返回块级元素列表。

    按 body 顺序遍历段落和表格,按真实交错顺序编号。

    Args:
        path: 文档文件的绝对路径。

    Returns:
        块级元素列表,每个元素含 id / type / text 等字段。

    Raises:
        FileNotFoundError: path 指向的文件不存在。
    """
```

**注释**:

- 注释写"为什么",不写"是什么"。代码已经说明是什么,注释要解释为什么这样做
- 反例:`i += 1  # i 加 1`
- 正例:`i += 1  # 跳过表头行`

### 1.4 Pythonic 惯用法

以下是 Python 社区共识的"地道写法",优先使用:

- **路径处理**:用 `pathlib.Path`,不用 `os.path` 拼字符串
- **资源管理**:用 `with` 语句,不手动 close
- **推导式**:简单变换用列表/字典推导,复杂逻辑用显式循环
- **早返回**:减少嵌套,处理边界条件后尽快返回
- **类型标注**:对外接口(函数签名、类属性)必须有类型标注;内部辅助函数可选

类型标注是给人看的文档,不是给运行时用的。

### 1.5 通用工程原则

**DRY(Don't Repeat Yourself)**

同一段逻辑出现两次,视为巧合;出现三次,应抽象成函数或模块。
注意:是**逻辑重复**才抽象,不是**代码长得像**就抽象。两段代码
长得像但表达的是不同意图时,保持分开。

**SRP(Single Responsibility Principle)**

一个函数只做一件事。判断标准:能不能用一句简单的话描述它的作用,
如果描述时出现"并且"、"然后"、"同时",通常要拆。

**YAGNI(You Aren't Gonna Need It)**

不为"未来可能的需求"提前写代码。只实现当下真正用到的功能。
过度工程化的代价是维护成本,不是资产。

**显式优于隐式(Explicit over Implicit)**

- 函数参数写清楚,不依赖全局变量
- 配置字段写清楚,不靠约定俗成
- 异常类型写清楚,不用 `except Exception` 一把捞

出自 Python 之禅(`import this`),是 Python 社区最基础的美学。

## 第二部分:项目工程原则(L2 层)

本项目特有的工程哲学,以抽象原则表达。具体落地形式见第三部分契约。

### 2.1 解耦优先,避免过度解耦【纲领】

这是本项目最核心的工程原则。项目长期可维护性的根基在于解耦——
不同关注点之间互不干扰,改一处不影响其他处。

**解耦的五个维度:**

#### 维度 A:变化频率不同的东西应该分离

变化频率不同的关注点,放在一起会互相拖累。慢变的被快变的污染,
快变的被慢变的束缚。

典型表现:参数经常变(路径、阈值、模型名),业务逻辑不常变。
两者应该分开存储。

#### 维度 B:可执行实体应与触发方式分离

做事的东西(业务逻辑)和启动它的东西(调度/UI/定时器)应该分开。
实体本身能独立运行,触发方式只是多一种调用途径。

这条的价值:触发方式坏了,实体还能用;实体不依赖某种特定触发方式,
调试和迁移都方便。

#### 维度 C:单个失败应与整体流程分离

批量处理中,单次失败要被捕获并记录,不能让整个流程崩溃。
失败也要落盘,标明"失败"状态,重跑时能精确定位。

理由:失败是常态(外部服务抽风、输入异常、网络超时),要把失败
当成正常分支处理,不是异常情况。

#### 维度 D:通用工具应与业务逻辑分离

通用能力(读文件、调 API、写 Excel)和业务逻辑(怎么评估规范、
怎么分类邮件)应该分开存储。

判断方法:这段代码搬到另一个完全不同的项目能不能直接用?
能→是通用工具;不能→是业务逻辑。

理由:业务会变,通用工具不变。混在一起会导致通用工具被业务污染,
最终难以复用。

#### 维度 E:数据资产应与操作代码分离

prompt、schema、mapping 这类"数据"虽然以代码形式存在,
但本质是资产。它们应该和"操作它们的代码"分开。

理由:数据类资产的修改者可能不是程序员(业务方可能直接改 prompt)。
分开后改数据不改代码,降低修改门槛和风险。

---

**解耦的边界:不追求形式干净**

解耦的目的是**降低未来的维护成本**,不是追求形式上的"干净"。
当解耦带来的认知成本高于它减少的维护成本时,不要解耦。

反例:

- 一个 50 行的脚本,参数只有一个输入路径,你非要拆出
  配置文件 + 配置加载器——收益不抵成本
- 两个模块各用一次某个工具函数,你提前抽成共享工具——
  违反 YAGNI,等第三处要用时再抽也不迟

**判断原则**:解耦应该让你**省心**,不是让你**省事**。
省心 = 未来维护容易;省事 = 当下少写代码。两者冲突时选省心。

### 2.2 文档服务于接手者,不是作者

写文档前问三个问题:

1. 三个月后的我打开这份文档,能不能快速回忆起项目状态?
2. 一个新人(或新对话的 AI)读完,能不能立刻上手?
3. 有没有具体步骤,还是只有空话?

如果回答不了,说明文档没写到位。

**禁止写入文档的内容:**

- 空洞的口号或金句(最高原则是唯一例外,它是整个项目的纲领)
- 未来可能会有的东西("未来可能支持 XX")
- 代码能直接看出的信息("本函数接收一个字符串参数")

**应该写入文档的内容:**

- 目录结构和每层职责
- 文件之间的依赖关系(谁 import 谁)
- 非直觉的设计决策及其原因("为什么选 A 不选 B")
- 新增功能的标准步骤
- 已知的坑和解决方法

### 2.3 架构从实例中长出,不从想象中设计

不要为"未来可能的需求"提前设计架构。架构应该从**至少两个真实实例**
中提炼,不是凭空构造。

具体规则:

- 新增一层抽象:必须有至少两个真实用例在用,不允许"为未来预留"
- 新增一个通用工具:第二处要用时再抽,不提前抽
- 新增一个配置字段:当前有明确需求再加,不加"将来可能用到"的字段

**过度设计的典型症状**:

- 设计了 N 层架构,实际写代码时只用到其中 2-3 层
- 定义了复杂的数据契约,却只有一方在用(没有第二方对接)
- 预留了大量扩展点,但每个扩展点都没被用过

**例外**:如果某个抽象**现在不加、后面加的成本会指数级上升**
(典型如数据库 schema、对外 API 格式),可以提前设计。
但这类例外要能明确说出"为什么不提前做成本会爆炸",说不出就不是例外。

## 第三部分:项目内部契约

本节把第二部分的抽象原则落地为具体的边界约定。
契约只讲**什么能做、什么不能做**;**当前有什么、叫什么、怎么长**,
见 ARCHITECTURE.md。

**契约修改原则**:契约稳定,两边才能各自演化。尽量只加不改。
必须改时要评估对所有现存任务的影响。

### 3.1 shared 边界

**规则**:shared 存放零业务的通用工具。

**红线**:

1. 不允许包含业务词汇(函数名、常量、注释、文档皆是)
2. 不允许 import 任何任务代码(单向依赖:任务→shared,反之不行)
3. 函数只接受通用类型,不接受业务对象
4. 新增模块自问:搬到完全不同的项目能不能直接用?不能→不要进 shared

### 3.2 任务隔离

**规则**:每个任务是独立单元,彼此不相关。

**红线**:

1. 任务之间禁止相互 import
2. 任务只能依赖 shared,不允许依赖其他任务
3. 任务的入口脚本必须能独立运行(不依赖启动器)
4. 任务对外交互只有三种:读自己的配置、写输出文件、调 shared

### 3.3 配置与代码分离

**规则**:所有运行参数外置到配置文件,代码不含硬编码常量。

**红线**:

1. 路径、模型名、阈值、API Key 不允许出现在代码里
2. 配置按"读者和变化频率"拆分到不同文件,不能一个大 config 装所有
3. 配置字段必须有类型和用途声明(让启动器能生成可视化编辑 UI)
4. 配置字段必须遵守跨任务统一的命名约定(具体约定见架构文档 2.3)

### 3.4 启动器与任务的通信

**规则**:启动器与任务之间只通过文件和子进程通信,不通过 import。

**启动器侧红线**:

1. 禁止 import 任务代码
2. 禁止修改任务的业务逻辑(启动器不碰任务入口脚本)
3. 禁止绕过配置文件直接给任务传参

**任务侧约束**:

1. 任务必须保证启动器能实时看到输出(不能带缓冲)
2. 任务必须支持被启动器调用(接收任务目录参数)和独立运行
   (默认读自身目录配置)两种方式

具体实现方式见架构文档 2.5 和 3.3。

## 第四部分:判断原则

文档不可能覆盖所有情况。遇到没明确规定的新问题时,用本节的判断原则决策。

本节以 few-shot 形式给出典型场景的判断范例,供参考类比。

### 4.1 何时抽到 shared?

**判断标准**:至少两个真实用例已经用到,且该能力本身不含业务。

**范例**:

- ✓ 两个任务都要读 YAML 配置 → 抽 `shared/config_loader.py`
- ✓ 三个任务都要断点续跑逻辑 → 抽 `shared/io_utils.py`
- ✗ 只有一个任务用到某函数 → 不抽,留任务目录(违反 YAGNI)
- ✗ 函数名含"规范"、"邮件"等业务词 → 不抽(违反 shared 边界)
- ✗ 函数参数是业务对象 → 不抽,先抽象成通用类型再说

**反模式**:"我觉得以后可能会用到"是抽到 shared 最常见的错误理由。
未来的事交给未来,现在只处理已经发生的重复。

### 4.2 何时新增任务?

**判断标准**:业务本质不同,不是输入形态不同。

**范例**:

- ✓ 评估规范、分块标注、摘要 → 三个任务(业务目标完全不同)
- ✓ 规范入库(查重+上传+更新目录)→ 独立任务(是完整业务流程)
- ✗ 单份评估 vs 批量评估 → 同一任务(只是输入形态不同,业务一致)
- ✗ 用模型 A 评估 vs 用模型 B 评估 → 同一任务(模型是配置,不是业务)

**判断方法**:如果两件事的 prompt 完全一样,就是同一任务;
如果 prompt 需要完全重写,是不同任务。

### 4.3 何时加新依赖?

**判断标准**:标准库和现有依赖确实无法满足,且新依赖能内网搬运。

**范例**:

- ✓ 需要读 Excel → 加 `openpyxl`(现有依赖已有)
- ✓ 需要解析 YAML → 加 `PyYAML`(标准库无,能搬运)
- ✗ 为写个 CLI 加 `click` → 不加(argparse 够用)
- ✗ 为美化日志加 `rich` → 不加(print 够用,违反最高原则)
- ✗ 为写测试加 `pytest` + `pytest-mock` + ... → 慎重,项目规模小

**追加标准**:新依赖进项目前必须更新 requirements.txt,
并在 README 中记录搬运步骤。

### 4.4 何时打破上面的规范?

**判断标准**:最高原则(用户体验优先)与其他规范冲突时。

**范例**:

- ✓ 配置应外置,但某任务只有一个参数、改起来翻文件反而慢
   → 硬编码一个常量在 runner.py 顶部,加注释说明"单参数不值得外置"
- ✓ 函数应单一职责,但拆完用户要多点三次才能跑完任务
   → 合并回来,加注释说明"合并是为了操作连贯"
- ✓ 不该提前抽象,但某段代码每次改都要改五处、肯定还会改
   → 提前抽象,加注释说明"基于实际维护痛苦度提前抽象"
- ✗ 觉得写规范麻烦 → 不是理由,不能打破
- ✗ 觉得当前不重要以后再说 → 不是理由,以后更不会改

**执行要求**:每次打破规范都要**在代码注释或提交信息里简短说明理由**。
让未来的接手者知道这是有意识的决策,不是疏忽。

### 4.5 何时更新文档?

**判断标准**:改变会影响接手者判断时。

**范例**(必须更新):

- 新增/删除任务 → 更新架构文档 5.1
- 新增/删除 shared 模块 → 更新架构文档 1.3
- 改变目录结构 → 更新架构文档 1.1
- 做出非直觉的设计决策 → 追加到架构文档 4 节
- 新增项目级规则 → 更新哲学文档对应小节

**范例**(不必更新):

- 修 bug、小重构 → 改代码即可
- 任务内部的业务逻辑变化 → 更新任务的 README.md
- 配置字段默认值变化 → 改 config.yaml 即可

**文档滞后的代价**:三个月后接手者看到文档和代码对不上,
会不知道哪个是对的,最终谁也不信。维护文档是长期投资。

## 结尾:文档的使用与维护

### 本文档的读者

- **你自己**:遇到设计决策犯犹豫时回来翻
- **新对话的 AI**:每次新开对话时加载(完整版或精简版)
- **未来可能接手的同事**:上手项目前的必读
- **朋友与同行**:了解本项目方法论

### 本文档的维护

- **最高原则**:不变。这是项目的根
- **通用规范(L1)**:几乎不变。除非 Python 社区共识变化
- **项目工程原则(L2)**:少变。需要改时必须记录理由和日期
- **项目契约(L3)**:偶尔变。修改时评估对所有现存任务的影响
- **判断原则(L4)**:可追加。遇到新的典型场景就补充一条 few-shot

### 本文档的精简版

考虑到每次新对话都加载完整版会占用上下文,项目另备一份
`PHILOSOPHY_DIGEST.md`,只保留规则不保留解释,约 100-150 行。
日常 AI 协作加载精简版即可,深度讨论或新人 onboarding 才用完整版。

### 最后的话

**规范的目的是让你省心,不是增加负担**。

如果某天你发现某条规范每次都在拖慢你,说明它可能已经过时或本来就不对。
把它改掉,不要硬守。规范应该服务人,不是反过来。

文档也一样。如果文档变成了"写了但没人看"的负担,说明写得不对,
该精简。好文档是接手者主动愿意读的,不是被迫看的。
